The Big Picture

What does this tool actually do?

You're migrating a database from MySQL to MongoDB. This tool solves two hard problems using an LLM (Claude 3 Haiku on AWS Bedrock).

🔄

Feature 1 — Query Migration

You give it a SQL/JPA query. The LLM converts it into a MongoDB aggregation pipeline you can copy-paste directly.

🔍

Feature 2 — Validation

After migration, it finds data mismatches between MySQL and MongoDB and uses the LLM to diagnose why they happened.

🤖

The LLM Layer

Claude 3 Haiku on AWS Bedrock does the heavy lifting. Data never leaves your company's AWS VPC — no external API keys needed.

Before Any Endpoint

What Happens When the App Starts

Before any developer hits an endpoint, three registries load automatically via @PostConstruct. This is the caching strategy — no file reading or DB querying happens during actual requests.

📂 ProcessorRegistry.java

Walks the src/.../processors/ folder. Reads every *Processor.java file as a raw string. Stores in a map:

{ "OrderItemProcessor" → "full source code..." }

🗄️ SchemaRegistry.java

Connects to MySQL. Queries information_schema.COLUMNS for every table. Stores column metadata:

{ "orders" → "id | bigint | nullable: NO
 total | double | nullable: NO
 discount | double | nullable: YES" }

🍃 MongoSchemaRegistry.java

Connects to MongoDB. Samples one document from every collection. Stores field structure:

{ "orders" → "_id | Integer
 total | Double
 status | String
 items | ArrayList" }
⚡

Why load at startup? When a developer hits an endpoint, the service reads from these in-memory maps instantly — zero latency. No file I/O, no database connections during the actual request. This keeps response times fast and predictable.

Feature 1

Query Migration — /migrate-query

A developer has a JPQL query they need converted to MongoDB. They don't know the MongoDB equivalent. Here's exactly what happens when they hit the endpoint.

💡

The input query example: @Query("SELECT o FROM Order o JOIN o.items i GROUP BY o.id HAVING SUM(i.price) > :amount") — a JPA query that joins Orders with their Items, groups by order, and filters groups where total item price exceeds a threshold. This is non-trivial to translate by hand.

Step 1 — Developer sends request

POST  /api/migration-tool/migrate-query

The developer sends their SQL query and the target MongoDB collection name.

{
  "sqlQuery": "SELECT o FROM Order o JOIN o.items i GROUP BY o.id HAVING SUM(i.price) > :amount",
  "collectionName": "orders"
}
Step 2 — Controller receives it

MigrationToolController.java

The REST controller is the entry point. It does zero logic — just deserializes the JSON into a DTO and passes it to the service layer. Clean separation of concerns.

// Controller just routes — no business logic here
return migrationToolService.migrateQuery(request);
Step 3 — Service builds the LLM prompt

MigrationToolService.java — migrateQuery()

This is the brain. It constructs two carefully-crafted prompts to send to the LLM:

// System prompt — tells the LLM WHO it is
"You are a migration expert.
 Translate SQL/JPA to MongoDB aggregation pipeline.
 Return ONLY JSON with aggregationPipeline and explanation."

// User message — gives the LLM the TASK
"Target collection: orders
 Query: SELECT o FROM Order o JOIN o.items i GROUP BY o.id
        HAVING SUM(i.price) > :amount"
🎯

Why two prompts? The system prompt sets the LLM's behavior and output format. The user message contains the actual task. This separation ensures consistent, parseable JSON responses every time.

Step 4 — Bedrock call to Claude

LlmApiClient.java — AWS Bedrock

This class uses the AWS SDK BedrockRuntimeClient — no manual headers or API keys. Credentials come automatically from the EC2 instance's IAM role.

// What gets sent to AWS Bedrock
bedrockClient.invokeModel()
  model: "anthropic.claude-3-haiku-20240307-v1:0"

Request body:
  anthropic_version: "bedrock-2023-05-31"
  max_tokens: 1000
  system: <systemPrompt>
  messages: [ { role: "user", content: <userMessage> } ]

// Auth: EC2 IAM role — no hardcoded API keys anywhere
// Data stays inside AWS VPC — never leaves the network

// Response parsing:
extracts content[0].text   // (not choices[0].message.content like OpenAI)
returns raw string back to service
🔒

Why Bedrock, not OpenAI? Security. The prompts contain database schemas and processor source code — sensitive internal data. With Bedrock, everything stays inside the company's AWS VPC. No data ever touches external servers. Auth is handled by IAM roles, so there are zero API keys to manage or rotate.

Step 5 — LLM responds

Claude 3 Haiku returns the MongoDB equivalent

{
  "aggregationPipeline": "[
    { \"$unwind\": \"$items\" },
    { \"$group\": { \"_id\": \"$_id\", \"total\": { \"$sum\": \"$items.price\" } } },
    { \"$match\": { \"total\": { \"$gt\": amount } } }
  ]",
  "explanation": "JOIN translated to $unwind since items are embedded.
   GROUP BY becomes $group. HAVING becomes $match after $group."
}
📖

Translation breakdown: SQL JOIN → $unwind (since items are embedded in the document). SQL GROUP BY → $group. SQL HAVING → $match placed after $group (order matters in MongoDB pipelines).

Step 6 — Parse & return

MigrationToolService.java — back from LLM

objectMapper.readTree(rawResponse) parses the JSON string. If parsing fails (malformed LLM output), a fallback response returns the raw string so nothing is lost. The controller wraps it in ResponseEntity.ok() and sends HTTP 200.

Result — Developer gets the pipeline

Ready to copy into MongoRepository ✅

{
  "aggregationPipeline": "[{ \"$unwind\": \"$items\" }, { \"$group\": ... }, { \"$match\": ... }]",
  "explanation": "JOIN → $unwind. GROUP BY → $group. HAVING → $match after $group."
}
Prerequisite

The Spring Batch Validation Job

Before the /validate endpoint can work, a separate batch job must run after migration. This job populates the mismatch_log table.

🔄 Row-by-Row Comparison

The batch job reads every row from the MySQL orders table, finds the corresponding document in MongoDB by ID, and compares all fields between them.

✅ Fields Match

Nothing happens — the job moves to the next row.

❌ Fields Mismatch

A record is written to the mismatch_log MySQL table with both the SQL row and MongoDB document.

📋 What a mismatch_log record looks like

{
  "tableName":      "orders",
  "collectionName": "orders",
  "sqlRow":         "{ id: 101, total: 250, discount: 25, status: SHIPPED }",
  "mongoDocument":  "{ _id: 101, total: 250, status: SHIPPED }",     // ← discount is MISSING
  "detectedAt":     "2024-01-15T10:30:00"
}
⚠️

Key detail: Notice the MySQL row has a discount field but the MongoDB document doesn't. This is exactly the kind of mismatch the /validate endpoint will diagnose using the LLM.

Feature 2

Validation — /validate

The batch job found mismatches. Now the developer wants to know why. This endpoint feeds the LLM all the context it needs to diagnose the root cause.

Step 1 — Developer sends request

POST  /api/migration-tool/validate

{
  "tableName":          "orders",
  "collectionName":     "orders",
  "processorClassName": "OrderItemProcessor"   // the Spring Batch processor that did the conversion
}
Step 2 — Controller → Service

MigrationToolController.java → MigrationToolService.java

Same pattern as Feature 1 — controller just routes to the service's validate() method.

Step 3 — Service gathers 4 inputs

The service collects everything the LLM needs to diagnose

This is where the validate flow gets powerful. The service pulls data from 4 different sources and combines them into a single LLM prompt:

Input 1 — Database

Mismatch Samples

Top 5 most recent mismatches from the mismatch_log table via MismatchLogRepository. If empty → returns early with "run batch job first".

Input 2 — Registry

Processor Source Code

Full Java source of OrderItemProcessor from ProcessorRegistry — already in memory, no file reading.

Input 3 — Registry

MySQL Schema

Column metadata (types, nullability, defaults) from SchemaRegistry — loaded at startup.

Input 4 — Registry

MongoDB Schema

Field names and types inferred from sample docs, from MongoSchemaRegistry — loaded at startup.

🧠

Why all 4 inputs? The LLM needs the full picture. The mismatch samples show what went wrong. The processor code shows how data was transformed. The schemas show the expected structure on both sides. Without any one of these, the LLM would be guessing.

Step 4 — Bedrock call with all 4 inputs

LlmApiClient.java — same Bedrock call pattern

The system prompt constrains the LLM to classify the root cause as one of: CODE (bug in processor), DB_DATA (null/missing data in source), or SCHEMA_DESIGN (structural mismatch between SQL and Mongo schemas). All 4 inputs sent in the message body. Data stays within AWS VPC.

bedrockClient.invokeModel()
  model: "anthropic.claude-3-haiku-20240307-v1:0"
  auth: IAM role — no API keys
  body: all 4 inputs combined into user message
Step 5 — LLM diagnoses the root cause

Claude 3 Haiku analyzes all inputs together and returns:

{
  "mismatchPattern": "discount field missing across all 5 samples",
  "likelyCause":     "discount column is nullable YES in MySQL.
                      Processor does not handle null — field silently dropped.",
  "suggestedFix":    "mongoOrder.setDiscount(
                        sqlOrder.getDiscount() != null
                          ? sqlOrder.getDiscount()
                          : 0.0
                      );",
  "severity":        "HIGH",
  "rootCauseType":   "DB_DATA"
}
Step 6 — Developer takes action

Targeted fix based on rootCauseType ✅

Because rootCauseType is DB_DATA (not CODE or SCHEMA_DESIGN), the developer knows to go straight to the processor and add a null check — one fix, one place. No need to comb through schema designs or look for mapping bugs.

📡 Controller

Thin REST layer. Receives HTTP requests, deserializes JSON into DTOs, calls the service, and returns the response. No business logic.

🧠 Service

The orchestrator. Gathers inputs from registries and DB, builds LLM prompts, calls LlmApiClient, parses the response, handles fallbacks.

🤖 LlmApiClient

Uses AWS SDK BedrockRuntimeClient. Auth via EC2 IAM role — no API keys. Calls Claude 3 Haiku and extracts content[0].text from the response.

📚 Registries

Three in-memory maps loaded at app startup via @PostConstruct. They hold processor source code, MySQL column metadata, and MongoDB field structures. No I/O at request time.

🗃️ MismatchLogRepository

A Spring Data JPA repository. Reads from the mismatch_log MySQL table — populated by the Spring Batch job, consumed by the validate endpoint.

📦 DTOs

Simple Java objects that define the shape of HTTP request and response bodies. MigrateQueryRequest, MigrateQueryResponse, ValidateRequest, ValidateResponse.

Key Differences

OpenAI vs AWS Bedrock

We switched from OpenAI to AWS Bedrock. Here's exactly what changed and why.

OpenAI (old) AWS Bedrock (current)
Auth API key in Bearer header EC2 IAM role — no keys needed
Client RestTemplate HTTP call AWS SDK BedrockRuntimeClient
Model gpt-4o-mini anthropic.claude-3-haiku
Request format messages[] with role/content Same + anthropic_version field
Response format choices[0].message.content content[0].text
Data location OpenAI external servers Inside company AWS VPC
Why we switched — Security — schemas and source code never leave the network
🛡️

The core reason for switching: The prompts sent to the LLM contain database schemas, processor source code, and actual data samples from production databases. With OpenAI, this data left the company network. With Bedrock, everything stays inside the AWS VPC — same security boundary as the databases themselves. IAM roles also eliminate the risk of leaked API keys.
